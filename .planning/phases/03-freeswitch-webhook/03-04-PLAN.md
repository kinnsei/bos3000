**Then modify callback/statemachine.go:**

Read the existing statemachine.go first to understand the current flow. Then add three concerns:

1. **Recording triggers (REC-01):**
   - After CHANNEL_BRIDGE event handler transitions to "bridged" state:
     - Call `fsClient.StartRecording(ctx, aUUID, callID, "a")` for A-leg
     - Call `fsClient.StartRecording(ctx, bUUID, callID, "b")` for B-leg
     - Update callback_calls SET recording_status='recording'
     - If StartRecording fails, log error but do NOT abort the call (recording is best-effort)
   - Before finalization (on hangup or timeout):
     - Call `fsClient.StopRecording(ctx, aUUID, callID, "a")`
     - Call `fsClient.StopRecording(ctx, bUUID, callID, "b")`
     - Errors logged but not blocking

2. **Webhook triggers (HOOK-01):**
   - Create a helper function `publishWebhook(ctx, call *CallRecord)`:
     - Look up user's webhook_url and webhook_secret from users table (via sqldb.Named("auth"))
     - If webhook_url is empty, skip (customer has no webhook configured)
     - Build WebhookPayload from current call state (populate ALeg, BLeg, Bridge details)
     - Call webhook.CreateAndPublishWebhook(ctx, callID, userID, webhookURL, secret, payload)
     - Errors logged but not blocking (webhook is fire-and-forget from state machine perspective)
   - Call `publishWebhook` on EVERY status transition:
     - initiating (after call record created)
     - a_dialing (after originate A sent)
     - a_connected (on CHANNEL_ANSWER for A)
     - b_dialing (after originate B sent)
     - bridged (on CHANNEL_BRIDGE)
     - finished (on normal completion)
     - failed (on any failure path)
   - This covers all 7 statuses per CONTEXT.md decision

3. **Orphan event handling (per CONTEXT.md):**
   - In the ESL event dispatch handler, if a received event's callID is not found in active calls map:
     - Log warning with UUID, event name, and event details
     - Do NOT panic, do NOT block
     - This was specified in CONTEXT.md locked decisions
