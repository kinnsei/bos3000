**webhook/deliver.go:**
- Pub/Sub subscription with MaxRetries=0 (NO automatic Encore retry -- we handle retries ourselves via the cron poller to honor exact intervals from CONTEXT.md):
  ```go
  var _ = pubsub.NewSubscription(
      WebhookTopic, "webhook-delivery-worker",
      pubsub.SubscriptionConfig[*WebhookEvent]{
          Handler: pubsub.MethodHandler((*Service).DeliverWebhook),
          RetryPolicy: &pubsub.RetryPolicy{
              MaxRetries: 0,
          },
      },
  )
  ```
- `DeliverWebhook(ctx context.Context, event *WebhookEvent) error`:
  1. Update delivery status to 'delivering', increment attempt_count, set last_attempt_at
  2. Compute signature: `SignPayload(event.Secret, event.Payload)`
  3. Create HTTP POST request to event.WebhookURL with:
     - Body: event.Payload
     - Headers: Content-Type: application/json, X-Signature: {signature}, X-Delivery-ID: {deliveryID}
  4. HTTP client with 10s timeout
  5. On 2xx response: update status='delivered', set delivered_at, return nil
  6. On failure (network error or non-2xx):
     - Read up to 1KB response body (if available) for error context
     - Look up current attempt_count from DB
     - Call `retryIntervalForAttempt(attempt_count)` to get next interval
     - If interval > 0: update status='retrying', set next_retry_at=NOW()+interval, set last_error. Return nil (NOT error -- we don't want Pub/Sub retry, the cron poller handles it)
     - If interval == 0 (attempt >= 6): update status='dlq', set dlq_at=NOW(), set last_error. Return nil
  7. IMPORTANT: Always return nil to Pub/Sub handler. Retry scheduling is managed by DB + cron, NOT Pub/Sub retry mechanism. This is required because Encore's RetryPolicy uses automatic exponential doubling which cannot match the exact 30s/1m/5m/15m/1h intervals locked in CONTEXT.md.
- IMPORTANT: Do NOT log event.Secret (per anti-pattern from RESEARCH.md)

**webhook/retry.go:**
- Define exact retry intervals per CONTEXT.md locked decision:
  ```go
  // RetryIntervals defines the exact backoff schedule per CONTEXT.md decision.
  // Encore Pub/Sub RetryPolicy uses automatic exponential doubling which cannot
  // match these exact intervals, so we implement custom retry via cron polling.
  var RetryIntervals = []time.Duration{
      30 * time.Second,  // attempt 1
      1 * time.Minute,   // attempt 2
      5 * time.Minute,   // attempt 3
      15 * time.Minute,  // attempt 4
      1 * time.Hour,     // attempt 5
  }

  func retryIntervalForAttempt(attempt int) time.Duration {
      if attempt < 1 || attempt > len(RetryIntervals) {
          return 0 // signals DLQ
      }
      return RetryIntervals[attempt-1]
  }
  ```
- Declare cron job for retry polling (every 10 seconds to provide reasonable latency):
  ```go
  var _ = cron.NewJob("webhook-retry-poll", cron.JobConfig{
      Title:    "Poll webhook deliveries due for retry",
      Every:    10 * cron.Second,
      Endpoint: PollRetries,
  })

  //encore:api private
  func (s *Service) PollRetries(ctx context.Context) error
  ```
- `PollRetries` implementation:
  1. Query: `SELECT id, webhook_url, payload FROM webhook_deliveries WHERE status='retrying' AND next_retry_at <= NOW() ORDER BY next_retry_at ASC LIMIT 50`
  2. For each row: look up user's current webhook_secret from users table (via sqldb.Named("auth"))
  3. Publish WebhookEvent to WebhookTopic with delivery ID, URL, secret, payload
  4. Update status to 'pending' (it will transition to 'delivering' when handler picks it up)
  5. Log count of re-published deliveries
  6. Return nil (individual publish errors logged but don't fail the batch)

**webhook/migrations/1_create_webhook_deliveries.up.sql:**
```sql
CREATE TABLE webhook_deliveries (
    id BIGSERIAL PRIMARY KEY,
    call_id VARCHAR(64) NOT NULL,
    user_id BIGINT NOT NULL,
    event_type VARCHAR(30) NOT NULL,
    status VARCHAR(20) NOT NULL DEFAULT 'pending' CHECK (status IN ('pending', 'delivering', 'delivered', 'retrying', 'failed', 'dlq')),
    webhook_url TEXT NOT NULL,
    payload JSONB NOT NULL,
    attempt_count INT NOT NULL DEFAULT 0,
    max_attempts INT NOT NULL DEFAULT 5,
    next_retry_at TIMESTAMPTZ,
    last_attempt_at TIMESTAMPTZ,
    last_error TEXT,
    delivered_at TIMESTAMPTZ,
    dlq_at TIMESTAMPTZ,
    dlq_retry_count INT NOT NULL DEFAULT 0,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_webhook_user_created ON webhook_deliveries(user_id, created_at DESC);
CREATE INDEX idx_webhook_status ON webhook_deliveries(status) WHERE status IN ('pending', 'retrying', 'dlq');
CREATE INDEX idx_webhook_retry_poll ON webhook_deliveries(next_retry_at) WHERE status = 'retrying';
CREATE INDEX idx_webhook_call ON webhook_deliveries(call_id);
```
Note: Added `next_retry_at` column and `idx_webhook_retry_poll` index for the cron-based retry poller.
